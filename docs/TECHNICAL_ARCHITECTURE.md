# The Taco's Egg — Technical Architecture

## Overview

A browser-based text adventure with an 80s CRT terminal aesthetic. The player is a sysadmin at **ZeroOne Hosting** (https://zeroone.traefikturkey.com/) — a self-deprecating hosting company operating out of a garage in Kelso, WA. Staff subsists on Taco Bell. Backup power is a Hyundai Kona EV with a very long extension cord.

The player investigates a $0.75 compute time discrepancy that escalates into a cyber-espionage case. All interaction happens through a terminal emulator UI presenting as a ZeroOne Hosting shell session. The terminal prompt is `zeroone>`. Narrative responses are generated by a LangGraph JS multi-agent backend.

---

## World Context

All agent system prompts and narrative generation are grounded in this setting. Treat these as canonical facts the GameMaster and NPC nodes must stay consistent with.

**ZeroOne Hosting** — https://zeroone.traefikturkey.com/
- A small shared-hosting operation run from a two-car garage in Kelso, WA
- Brand voice: self-aware, self-deprecating, absurdist — knows it's a garage operation and leans into it
- Staff count: small enough that everyone knows everyone, which makes the espionage investigation personal
- Staff nutrition plan: Taco Bell, exclusively and enthusiastically
- Backup power: one Hyundai Kona EV parked outside with an extension cord running through a hole in the drywall. The UPS runtime depends on how much the Kona was driven that week.
- The servers are real. The threat is real. The ambiance is surreal.

**Terminal identity**
- Prompt: `zeroone> `
- Login banner should feel like a real (slightly broken) hosting company — uptime stats, a passive-aggressive note about disk quotas, maybe a Taco Bell order reminder
- Locations the player can navigate: server room (the back half of the garage), break room (a folding table), the parking lot (where the Kona lives), the NOC (a desk with three monitors and a lava lamp)

**Tone guidance for agents**
- Dry humor is correct. Winking absurdism is correct. Slapstick is not.
- The cyber-espionage plot is played straight — the comedy comes from the contrast with the setting, not from undercutting the stakes.
- NPCs have opinions about menu items. This is load-bearing characterization.

## 1. Frontend Architecture

### SvelteKit App Structure

The SvelteKit app serves two concerns: the game UI and a thin API route that forwards commands to the agent backend.

`src/routes/+layout.svelte` sets up the full-viewport black background and mounts `CRTOverlay`. The game page (`+page.svelte`) mounts the `Terminal` component and nothing else — no navigation, no modals, no chrome.

### CRT Terminal Component

```
Terminal.svelte
├── CRTOverlay.svelte     (CSS layer: scanlines, vignette, flicker)
├── TerminalOutput.svelte (output buffer, auto-scroll, text rendering)
└── TerminalInput.svelte  (prompt, input field, history navigation)
```

**Input handling:**
- Single `<input>` element, always focused, captured at the `Terminal` level
- Prompt prefix is `zeroone> ` — rendered as static text before the input field
- `Enter` submits the current line to the command parser
- `ArrowUp` / `ArrowDown` cycle through command history (stored in `terminal` store)
- `Ctrl+C` cancels any pending streaming response via `AbortController`

**Command parsing (`src/lib/game/commands.ts`):**

```
raw input string
  → tokenize (split on whitespace, handle quoted strings)
  → route: check against local command table first
      - local commands (HELP, CLEAR, HISTORY) handled client-side, no network call
      - all others forwarded to POST /api/command
  → append user line to output buffer
  → await streamed response, append tokens to output buffer
```

**Output rendering:**
- Output buffer is an array of `OutputLine` objects: `{ id, text, type, timestamp }`
- Types: `user-input | system | narrative | error | npc-dialogue | evidence`
- Each type maps to a CSS class for color (amber/green/red/white per 80s terminal convention)
- `narrative` lines render via a typewriter effect (token-by-token from the SSE stream)
- Evidence lines get a distinct prefix (`[EVIDENCE]`) and are mirrored to the evidence log

**CRT visual effects (`CRTOverlay.svelte`):**

Pure CSS — no WebGL or shader dependency.

### Game State Management (Client)

Three Svelte stores:

**`game.ts`** — read-only mirror of server-authoritative game state:
```typescript
interface GameState {
  chapter: number;
  inventory: string[];
  evidence: EvidenceItem[];
  meters: {
    suspicion: number;   // 0–100; NPCs become evasive above 60
    trust: number;       // 0–100; unlocks information at thresholds
    time: number;        // in-game minutes elapsed
  };
  flags: Record<string, boolean>;
  currentLocation: string;
  activeNPC: string | null;
}
```

**`terminal.ts`** — UI-only state (not persisted):
```typescript
interface TerminalState {
  outputBuffer: OutputLine[];
  inputHistory: string[];
  historyIndex: number;
  isStreaming: boolean;
  streamAbortController: AbortController | null;
}
```

**`session.ts`** — session identity:
```typescript
interface SessionState {
  sessionId: string;   // UUID, generated on first load, persisted to localStorage
}
```

The `game` store is updated from each API response. The client never directly mutates `GameState`; it sends commands and applies the state delta returned by the server.

## 2. AI / Agent Architecture (LangGraph JS)

### Graph Overview

```
                    ┌─────────────────┐
user command ──────►│    Terminal     │
                    │  Interpreter   │
                    └────────┬────────┘
                             │ classified intent + entities
                             ▼
                    ┌─────────────────┐
                    │   GameMaster    │◄──── Redis checkpoint
                    └──┬──────┬───┬───┘
                       │      │   │
          ┌────────────┘      │   └──────────────┐
          ▼                   ▼                  ▼
   ┌────────────┐   ┌──────────────────┐  ┌───────────────┐
   │    NPC     │   │  Investigation   │  │   TacoShop    │
   │  Dialogue  │   │    Tracker       │  │    System     │
   └─────┬──────┘   └────────┬─────────┘  └──────┬────────┘
         │                   │                   │
         └───────────────────┴───────────────────┘
                             │ merged output
                             ▼
                  ┌──────────────────────┐
                  │   Response Formatter │
                  └──────────┬───────────┘
                             │ streamed tokens + state delta
                             ▼
                           client
```

### Agent Nodes

**TerminalInterpreter**
- First node in every invocation
- Classifies raw input into: `navigation | examine | inventory | dialogue | taco-order | meta | unknown`
- Extracts entities: target object, NPC name, item name
- Model: GPT-4o-mini — latency matters here more than capability

**GameMaster**
- Central routing node; owns narrative continuity
- Reads full graph state (location, chapter, flags, meters)
- Uses conditional edges to route to specialist nodes based on classified intent:
  - `dialogue` intent + `activeNPC` set → NPC node
  - `taco-order` intent → TacoShop node
  - evidence-adjacent intent → Investigation node
  - everything else → handled directly, then ResponseFormatter
- Generates environmental narrative when no specialist is needed
- Updates chapter progression and story flags after specialists return
- Model: GPT-4o

**NPC**
- Maintains per-NPC voice, knowledge state, and disposition
- Reads suspicion/trust meters to modulate what the NPC reveals
- Has access to sliding-window NPC conversation history (last 20 exchanges)
- Output: dialogue lines tagged with the NPC name
- Model: GPT-4o

**Investigation**
- Tracks clues, evidence, and logical deductions
- Validates whether a player action constitutes discovering evidence
- Updates evidence inventory when triggered
- Can be queried directly when player types `REVIEW EVIDENCE`
- Returns structured data to GameMaster for narration — does not generate prose itself
- Model: GPT-4o-mini (structured output task)

**TacoShop**
- Handles the Taco Bell ordering mechanic — the de facto staff lunch program at ZeroOne Hosting
- Maintains order state (items ordered, running cost, pending items)
- Ordering food unlocks certain NPC dialogue, fuels late-night debugging sessions, and serves as cover story in several chapters
- The absurdist premise (investigating espionage between Gordita runs) is a first-class narrative element, not a throwaway bit
- Model: GPT-4o-mini

### Graph State Schema (`server/state/schema.ts`)

```typescript
import { z } from "zod";

export const GraphStateSchema = z.object({
  // Input
  sessionId: z.string(),
  rawInput: z.string(),
  classifiedIntent: z.string().optional(),
  extractedEntities: z.record(z.string()).optional(),

  // Game world
  chapter: z.number(),
  location: z.string(),
  activeNPC: z.string().nullable(),
  inventory: z.array(z.string()),
  evidence: z.array(z.object({
    id: z.string(),
    description: z.string(),
    discoveredAt: z.number(),   // in-game timestamp
    chapter: z.number(),
  })),
  flags: z.record(z.boolean()),
  meters: z.object({
    suspicion: z.number().min(0).max(100),
    trust: z.number().min(0).max(100),
    time: z.number(),
  }),

  // Output accumulation
  narrativeChunks: z.array(z.string()),
  stateDelta: z.record(z.unknown()).optional(),

  // Routing
  nextNode: z.string().optional(),
  error: z.string().optional(),
});

export type GraphState = z.infer<typeof GraphStateSchema>;
```

### Conditional Edges

GameMaster uses a routing function as a LangGraph conditional edge:

```typescript
function routeFromGameMaster(state: GraphState): string {
  if (state.error) return "error_handler";
  if (state.classifiedIntent === "taco-order") return "taco_shop";
  if (state.classifiedIntent === "dialogue" && state.activeNPC) return "npc";
  if (state.classifiedIntent === "examine" && isEvidenceContext(state)) return "investigation";
  return "response_formatter";
}
```

All specialist nodes route unconditionally to `response_formatter` on completion. The formatter assembles the final streamed response and the state delta.

### Memory / State Persistence (Redis)

LangGraph JS supports custom checkpointers. Each graph invocation checkpoint is keyed by `sessionId`.

```
Redis key structure:
  checkpoint:{sessionId}:latest      → serialized GraphState (full)
  npc:{sessionId}:{npcName}:memory   → NPC conversation history (last 20 turns)
  session:{sessionId}:meta           → created_at, last_seen, total_play_seconds
```

The Redis checkpointer (`server/memory/checkpointer.ts`) implements the LangGraph `BaseCheckpointSaver` interface. Each API call resumes from where the previous one left off without the client sending game state back up.

NPC memory uses a sliding window (last 20 exchanges) to keep token usage bounded while preserving conversational continuity within a session.

## 3. Game State Management

### What State Lives Where

| State                         | Primary Location              | How Client Gets It                   |
|-------------------------------|-------------------------------|---------------------------------------|
| Chapter, location, flags      | Redis (LangGraph checkpoint)  | `state_delta` SSE event per response |
| Inventory, evidence           | Redis                         | `state_delta` SSE event per response |
| Meters (suspicion/trust/time) | Redis                         | `state_delta` SSE event per response |
| NPC conversation history      | Redis                         | Internal to agent graph only          |
| Output buffer                 | Client memory only            | Not persisted                         |
| Command history               | localStorage                  | Session-scoped                        |
| Session ID                    | localStorage                  | Stable across browser sessions        |

The client `game` store is a read-only projection of what the server last returned. No game logic executes client-side.

### Save / Load

There is no explicit save/load mechanic — Redis state IS the save. When a player returns:

1. Browser reads `sessionId` from localStorage
2. First command triggers a graph invocation with that `sessionId`
3. LangGraph checkpointer restores full `GraphState` from Redis
4. GameMaster emits a returning-player narrative

For explicit named save slots (future): add a `slot` suffix to the checkpoint key and expose `SAVE [name]` / `LOAD [name]` commands. The current architecture requires no changes to support this.

Redis TTL is 90 days on checkpoint keys, reset on each access.

## 4. API Design

### Endpoints

```
POST /api/command
  Body:    { sessionId: string, input: string }
  Returns: text/event-stream (SSE)

GET  /api/session/new
  Returns: { sessionId: string }
  Called once on first load if localStorage has no session ID.
```

### Command to Response Flow

```
1.  Player presses Enter
2.  Terminal.svelte reads input, appends [user-input] line to output buffer
3.  api/client.ts opens SSE connection: POST /api/command
4.  SvelteKit +server.ts receives request, validates sessionId
5.  Passes { sessionId, input } to graph.stream()
6.  Graph executes: TerminalInterpreter → GameMaster → [specialist] → ResponseFormatter
7.  ResponseFormatter streams narrative tokens via LangGraph streaming events
8.  +server.ts pipes LangGraph stream to the SSE response
9.  Client receives SSE chunks, appends tokens to output buffer in real-time
10. On stream close, client reads the final state_delta event and updates game store
```

### SSE Event Format

```
event: token
data: {"text": "The ZeroOne Hosting terminal flickers. The Kona hums outside."}

event: token
data: {"text": " A new file appears in /var/log/zeroone."}

event: state_delta
data: {"chapter": 2, "flags": {"found_logfile": true}, "evidence": [...]}

event: done
data: {}
```

`state_delta` carries only changed fields. The client merges this into the game store rather than replacing it wholesale.

### SvelteKit Route (`src/routes/api/command/+server.ts`)

```typescript
export async function POST({ request }): Promise<Response> {
  const { sessionId, input } = await request.json();

  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    async start(controller) {
      const send = (event: string, data: unknown) => {
        controller.enqueue(
          encoder.encode(`event: ${event}\ndata: ${JSON.stringify(data)}\n\n`)
        );
      };

      for await (const chunk of graph.stream(
        { sessionId, rawInput: input },
        { configurable: { thread_id: sessionId } }
      )) {
        if (chunk.token) send("token", { text: chunk.token });
        if (chunk.stateDelta) send("state_delta", chunk.stateDelta);
      }

      send("done", {});
      controller.close();
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
      "Connection": "keep-alive",
    },
  });
}
```

## 5. Deployment Considerations

### Environment Variables

```bash
# .env.example

# AI provider
OPENAI_API_KEY=

# Redis (Upstash recommended for serverless; or self-hosted)
REDIS_URL=
REDIS_TOKEN=          # Upstash only

# App
PUBLIC_SESSION_TTL_DAYS=90
NODE_ENV=development
```

No secrets reach the client bundle. The SvelteKit API route is the only code that touches `OPENAI_API_KEY` or `REDIS_URL`.

### Development Setup

SvelteKit dev server handles both the frontend and the `/api/command` route. The LangGraph graph runs in-process within the SvelteKit server. Redis can be a local Docker container:

```bash
docker run -d -p 6379:6379 redis:7-alpine
```

No separate backend process is needed in development. The `server/` directory is imported directly by the SvelteKit route handler.

### Production Setup

**Option A — Single SvelteKit Node deployment (start here)**
- Deploy to Fly.io or similar Node-capable host
- SvelteKit runs as a Node server (not edge — LangGraph requires Node APIs)
- Redis: Upstash (serverless Redis, free tier sufficient for early access)
- Adapter: `@sveltejs/adapter-node`

**Option B — Decoupled (if needed later)**
- SvelteKit frontend → static or edge deployment
- Separate Express/Hono server for the agent graph
- Only warranted if agent processing time regularly exceeds serverless function timeouts

Start with Option A.

```javascript
// svelte.config.js
```

### Graph Initialization (`server/agents/graph.ts`)

The graph is instantiated once at module load and imported by the SvelteKit route. The Redis checkpointer is initialized from `REDIS_URL` at startup.

```typescript
import { StateGraph } from "@langchain/langgraph";
import { RedisCheckpointer } from "../memory/checkpointer.js";
import { GraphStateSchema } from "../state/schema.js";

const checkpointer = new RedisCheckpointer(process.env.REDIS_URL!);

export const graph = new StateGraph(GraphStateSchema)
  .addNode("terminal_interpreter", terminalInterpreterNode)
  .addNode("game_master", gameMasterNode)
  .addNode("npc", npcNode)
  .addNode("investigation", investigationNode)
  .addNode("taco_shop", tacoShopNode)
  .addNode("response_formatter", responseFormatterNode)
  .addEdge("__start__", "terminal_interpreter")
  .addEdge("terminal_interpreter", "game_master")
  .addConditionalEdges("game_master", routeFromGameMaster)
  .addEdge("npc", "response_formatter")
  .addEdge("investigation", "response_formatter")
  .addEdge("taco_shop", "response_formatter")
  .addEdge("response_formatter", "__end__")
  .compile({ checkpointer });
```
